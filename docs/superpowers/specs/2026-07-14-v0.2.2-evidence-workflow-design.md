# v0.2.2 Evidence Workflow Design

## Goal

Add a customer-data-free regression and observed-usage workflow that improves confidence in Excel BI skill routing without representing synthetic fixtures, generated responses, or local logs as real workbook success.

## Scope

The release adds five deterministic sanitized case definitions and a local usage-evidence contract. It does not ship customer workbooks, capture telemetry automatically, upload data, alter the twelve published skill IDs, or add Excel product features.

## Architecture

The existing `fixtures/real-sanitized-cases/` library remains the source of regression problem shapes. Five new JSON cases extend coverage for Power Query lineage/refresh, DAX compatibility/dependency, CUBE zero-result diagnosis, VBA button binding, and deliverable/external-dependency cleanup.

Observed usage is a separate local JSONL stream. A standard-library validator reads one event per line, rejects unsafe paths, credentials, customer artifact names, and unsupported evidence claims, then emits a normalized report. A summarizer aggregates routing, completion, duration, and evidence-level counts without promoting structural or runtime evidence to workbook-behavior proof.

## Event Contract

Each observed-usage event is an object with:

- `schemaVersion`: `1.0`;
- `eventId`: unique opaque identifier;
- `recordedAt`: ISO-8601 UTC timestamp;
- `caseId`: a checked-in sanitized case ID or `local-user-supplied-sanitized`;
- `requestedSkill`, `selectedSkill`, `outcome`: routing and result fields;
- `durationMs`: non-negative integer;
- `evidenceLevel`: `structural`, `runtime-capability`, or `workbook-behavior`;
- `evidenceBoundary`: explicit statement that the event applies only to its recorded machine/workbook scope;
- `notes`: optional sanitized operational note.

`workbook-behavior` requires `local-user-supplied-sanitized`; it must never include a workbook path, file name, secret, customer name, or generated report payload. Invalid lines are reported with line numbers and do not produce a passing validation status.

## Commands

`validate_observed_usage.py` validates JSONL and can write a sanitized report. `summarize_observed_usage.py` consumes a valid JSONL file and produces JSON plus optional Markdown. Both tools default to explicit caller-selected local output paths; no log is written into the repository.

`run_case_regression.py` gains checks for the five added definitions. The structural release gate validates the checked-in case count and the usage-tool contracts only; it never requires a local usage log, opens Excel, or claims real workbook success.

## Evidence Rules

Synthetic case specs prove only reproducible problem shapes. Structural results prove file and contract checks. Runtime-capability evidence remains machine-specific. Workbook-behavior evidence is valid only for a deliberately sanitized, user-supplied workbook and its stated authoring/automation/consumer scope. Summaries retain these distinctions verbatim.

## Testing

Tests create temporary JSONL inputs for valid structural, runtime, and workbook-behavior events; malformed JSON; duplicate IDs; unsafe paths; secret-like text; invalid evidence transitions; and aggregation correctness. Fixture tests verify all twelve sanitized case IDs, each required layer, and no customer artifact extensions. Release-gate tests confirm the new workflow is structural-only.

## Non-Goals

This release does not collect real data automatically, invoke Excel during CI, create a hosted analytics service, publish observed usage, or lower the existing v0.2.1 static-risk grade by deleting detailed skill guidance.
