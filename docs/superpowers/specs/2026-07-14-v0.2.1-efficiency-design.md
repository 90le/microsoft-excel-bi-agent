# Microsoft Excel BI Agent v0.2.1 Efficiency Design

## Objective

Reduce discovery and invocation cost without changing the twelve published skill IDs or adding Excel features. The release improves starter-prompt usability, skill-trigger precision, and measurement evidence while preserving v0.2.0 routing and compatibility behavior.

## Chosen approach

Use a metadata-first optimization rather than rewriting every skill body or splitting the plugin into smaller products.

1. Shorten the three manifest starter prompts to no more than 110 characters each while retaining inspect, diagnose, and publish intent.
2. Rewrite each canonical skill description as a concise, trigger-only sentence beginning with `Use when`. Preserve domain keywords and confusable boundaries, but do not summarize the skill workflow.
3. Establish a checked-in benchmark configuration and trigger corpus before changing descriptions. Measure positive selection, confusable negative selection, token cost, tool-call success, and task completion evidence.
4. Keep detailed workflows, scripts, and references unchanged unless a failing benchmark proves that body text is the source of a problem.

## Alternatives considered

### Aggressive body compression

Rewrite all twelve `SKILL.md` bodies and move most guidance into references. This could reduce invocation tokens further, but it has a high regression risk and would require twelve independent behavior validations. It is deferred until observed usage shows that invoked skill bodies, rather than metadata, dominate cost.

### Split the plugin into multiple smaller plugins

Publish separate authoring, modeling, diagnostics, and delivery packs. This could reduce discovery cost for narrow users, but it introduces installation choices, duplicated routing, and compatibility migration. It is deferred until real usage demonstrates distinct user populations.

### Metadata-first optimization

This is the selected approach. It addresses the evaluator's strongest valid findings with the smallest compatibility surface and produces evidence needed for later architectural decisions.

## Scope

### Included

- Three shorter default prompts.
- Trigger-only descriptions for all twelve canonical skills.
- Baseline and post-change plugin-eval reports for the staged runtime package.
- A benchmark configuration with representative Excel BI scenarios.
- Positive and confusable-negative trigger cases for every skill.
- Cross-agent mirror synchronization after canonical changes.
- Version and release documentation for `v0.2.1`.

### Excluded

- New skill IDs or renamed skills.
- New Excel, Power BI, VBA, Power Query, DAX, MDX, or ADO functionality.
- Splitting `run_release_gate.py`; that is a separate maintainability release.
- Claims of certified behavior on Office versions not represented by live evidence.
- Publishing benchmark scores based only on generated or synthetic responses.

## Trigger contract

Each canonical `.agents/skills/<skill>/SKILL.md` description must:

- start with `Use when`;
- describe user-visible triggers and task context only;
- remain under 300 characters, with a target of 120 to 220 characters;
- retain the important product, file-format, and error vocabulary for that domain;
- avoid procedural steps that let an agent skip reading the skill body;
- remain distinguishable from adjacent skills in the benchmark corpus.

The published `skills/`, `.claude/skills/`, and `.opencode/skills/` trees remain generated mirrors and are not edited directly.

## Benchmark design

The checked-in benchmark uses customer-data-free prompts and separates three evidence types:

1. **Static budget evidence:** plugin-eval before/after reports for the allowlisted runtime package.
2. **Deterministic routing evidence:** existing router and release-gate fixtures must remain green.
3. **Fresh-agent behavior evidence:** positive and confusable-negative prompts are run in isolated sessions and scored for selected skill, useful result, tool discipline, and unsupported claims.

Synthetic responses may validate scorer mechanics but cannot count as real task-success evidence. Observed usage files and generated reports remain local artifacts unless deliberately sanitized.

## Success criteria

- All three default prompts are at most 110 characters.
- All twelve descriptions meet the trigger contract.
- Static `trigger_cost_tokens` decreases by at least 30 percent from the v0.2.0 runtime baseline of 1,161 tokens.
- Static `invoke_cost_tokens` does not increase.
- Every skill has at least two positive and one confusable-negative benchmark prompt.
- Existing 45 unit tests, skill validation, structural gate, mirror-drift check, and Windows localized-path CI remain green.
- No public claim converts synthetic benchmark output into real-world success evidence.

## Versioning and delivery

The work ships as `v0.2.1`. It starts from the released `v0.2.0` main branch, uses `agent/v0.2.1-efficiency`, and is published through a draft pull request. The installed global plugin remains on stable `v0.2.0` until v0.2.1 is reviewed and released.
