# Power Query M Style Guide

Read this file before writing or rewriting substantial M queries. Do not read it for small one-line formula fixes.

## Naming

- Use business-meaningful step names.
- Avoid fragile autogenerated names such as `Changed Type1` when delivering maintainable code.
- Use ASCII step names when toolchains may corrupt non-ASCII source. Preserve Chinese column names exactly when they are workbook business fields.

## Query Shape

Keep a stable progression:

```powerquery
let
    Source = ...,
    FilteredFiles = ...,
    ParsedRows = ...,
    NormalizedColumns = ...,
    JoinedLookup = ...,
    FinalColumns = ...
in
    FinalColumns
```

## Source Hygiene

- Filter hidden files and `~$` temp files early.
- Normalize extensions with `Text.Lower`.
- Keep `.xls`, `.xlsx`, `.xlsm`, and `.csv` handling explicit.
- Avoid hard-coded sample file row counts unless the source format is contractual.

## Type Handling

- Apply types after column normalization.
- Use `try ... otherwise null` for unstable external data.
- Prefer nullable types where blanks are valid.
- Avoid applying `Changed Type` before confirming headers.

## Joins

Before expanding a joined table:

1. Check lookup key uniqueness.
2. Pre-aggregate duplicate lookup rows if the main table must stay one row per original row.
3. Preserve original order if the output is report-facing.

## Delivery Layer

Do final renames, blank normalization, and column trimming at the end. Avoid mutating upstream source queries just to satisfy final report naming.
